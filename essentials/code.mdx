## Default example formats

- Use `bash` for request examples
- Use `json` for request and response payloads
- Use inline code for endpoints, headers, file names, and payload fields

## API example rules

- Show the real endpoint path and the real required headers
- Use lowercase request field names such as `name`, `keywords`, and `lang`
- Keep the first example minimal, then add optional fields only when they matter
- If the live response has legacy quirks, document them instead of normalizing them away

## Important live-contract quirks

- Job objects currently return mixed-case top-level keys such as `ID`, `Name`, and `Data`
- `Data.max_time` in job responses is serialized as a Go duration integer, not the original seconds input
- API key examples should use the `bscraper_` prefix

## Example request pattern

```bash
curl -X POST https://api.brezelscraper.com/api/v1/jobs \
  -H "Authorization: Bearer bscraper_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Berlin cafes",
    "keywords": ["cafes in berlin"],
    "lang": "en"
  }'
```
