## Voice and structure

- Lead with the user task, not the product feature
- Use active voice and second person
- Keep sentences short and concrete
- Use sentence case for headings
- Bold UI labels such as **Jobs**, **Credits**, and **Integrations**
- Use code formatting for commands, paths, endpoints, headers, and payload fields

## Product terminology

- Use `job` for a scraping run
- Use `results` for extracted business data
- Use `credits` for balance and usage
- Use `dashboard` for the authenticated app
- Use `integrations` for Google Sheets, API keys, and webhooks

## What to explain

- Explain how a setting changes cost, runtime, or result volume when that tradeoff matters
- Start with the dashboard workflow before the API workflow when both exist
- Show the happy path first, then call out common failure modes

## What to avoid

- Do not promise future features or unpublished contracts
- Do not copy stale OpenAPI or starter content without verifying the code
- Do not use vague synonyms when a product term already exists
